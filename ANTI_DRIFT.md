# Anti-Drift Guardrail

> **Read this at the start of every working session.** Before beginning any task,
> confirm it serves the current phase in `ROADMAP.md` and the thesis below. If a
> proposed change does not serve the core thesis, **flag it to the author instead
> of building it.**

## The project, in one sentence

Aletheia is a multi-agent verification framework, with a rigorous evaluation
harness, that improves the reliability of LLM-generated answers by grounding
every claim in real evidence and surfacing disagreement instead of hiding it.

The locked design decisions that shape every phase — domain focus, the
verification-not-advice safety boundary, the corpus-first knowledge source,
disagreement handling, explained confidence, and the benchmarking split — live as
Architecture Decision Records in [`docs/design/`](docs/design/). Re-read them
alongside this guardrail.

## Non-negotiables

1. **Free tier only.** Every tool, library, model, and host must have a genuinely
   free option. Never introduce anything that requires payment. Runtime LLM power
   uses free API tiers (Gemini, Groq) via user-supplied environment variables.
2. **Production quality only.** Every file, name, comment, test, and commit must
   read like the work of a senior engineer. No placeholder junk, no "TODO fix
   later" left lying around, no toy code.
3. **Author attribution only.** Git config, commits, README, and every document
   use Jay Gautam's identity. Never add "Generated with …", `Co-authored-by:` an
   automated agent, or any AI/tool attribution — in commits, code, or docs. Ever.
4. **The evaluation harness is the centerpiece.** Features exist to be measured.
   If a feature cannot be evaluated or does not move a headline metric, question
   whether it belongs.
5. **Build phase-by-phase.** Never one-shot. Plan, get approval, implement small
   reviewable steps, test, document, commit.
6. **Update GitHub every session.** Update the README (if status changed), append
   a dated plain-language entry to `PROGRESS_LOG.md`, tick `ROADMAP.md` boxes,
   then commit and push.

## OUT OF SCOPE — DO NOT DO

- ❌ Do **not** ever generate medical advice, diagnosis, treatment, or dosing
  recommendations. Aletheia **verifies whether a *claim* is supported by evidence** —
  it is a research tool, never a medical advisor. The "research tool — not medical
  advice; consult a professional" disclaimer must appear in the UI, every API response,
  and the README. (Hard non-negotiable — see
  [`docs/design/0002`](docs/design/0002-verification-not-medical-advice.md).)
- ❌ Do **not** turn this into a general-purpose chatbot or assistant.
- ❌ Do **not** add any paid service, paid API, or anything without a free tier.
- ❌ Do **not** expand to many domains. Keep the corpus and benchmarks focused;
  breadth dilutes the measurable result.
- ❌ Do **not** gold-plate features that do not serve verification + evaluation
  (no elaborate auth systems, billing, social features, multi-tenant dashboards,
  etc., unless a phase explicitly calls for it).
- ❌ Do **not** add AI/agent co-authorship or "generated by" lines anywhere.
- ❌ Do **not** skip the harness to chase a flashier demo. The table is the point.
- ❌ Do **not** introduce heavy frameworks where a small, legible solution wins.
- ❌ Do **not** silently swap a major architectural decision; justify it in
  writing and, when significant, ask the author first.

## Decision rule

When unsure whether something is in scope, ask:

> *Does this make the evidence-grounded verification more correct, the evaluation
> more rigorous, or the system more reliably runnable on free tiers?*

If **no**, it is probably drift. Flag it.
